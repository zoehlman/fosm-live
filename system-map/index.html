How the data flows— Zach feeds the Brain · chambers feed their layer · instances roll up to their layer's Apex
↑ 4 streams → Brain. Zach feeds frameworks & vision (the irreplaceable human intelligence). Client data feeds FOSM KPIs, financials, and what's working in the field. Industry data feeds sector trends and benchmarks. Economic data feeds macro context. The Brain triangulates all four — it's not "Zach's voice," it's an intelligence platform that gets smarter with every client added.
→ Chambers → Instances. The For-Profit chamber pushes to every for-profit instance. Non-Profit pushes to every foundation. Retreat pushes to every retreat location. Same architecture, three flavors of playbook — each instance always running the latest.
▲ Instances → 3 Apex CCs. For-profit instances roll up to the For-Profit Executive CC. Foundations roll up to the Non-Profit Executive CC. Retreat locations roll up to the Retreat Executive CC. Each apex shows the same RYG grid pattern — different content, same view of "is everything healthy?"
FOSM·LIVE Master — the BRAIN · AI · framework intelligence · pushes to every instance
★ fed by Zach ↔ feeds every FOSM·LIVE CC instance
This is the intelligence layer, not an instance. The Brain holds the frameworks, the voice, the playbooks, the proven moves — across three chambers (For-Profit, Non-Profit, Retreat) and continuously fed by four input streams (Zach, Client Data, Industry Data, Economic Data). Sharpen a framework here, and every relevant instance gets smarter at once. Add a new client, and the Brain learns from their data — every other instance benefits.
FOSM·LIVE = the business engine + the human engine. Most frameworks stop at F.O.S.M. (the company). LIVE adds what runs the human running the company. Both halves apply to companies, foundations, and retreat centers — the chambers just translate the vocabulary. Architecture notes for the back-end— so we build it right the first time
Tenant isolation by design. Row-level security on every table. A query for Tenant A can never return a row tagged Tenant B. Verified at the DB layer, not the app layer.
Shared schema, separate data. All tenants use the same FOSM·LIVE schema (so one upgrade benefits everyone). Their data lives in their tenant partition.
Roles & principals. Tenant admin · entity owner · entity contributor · read-only viewer. Permissions are inherited down the entity tree.
